FASE 01 🖥️ Monitor — Preparação do Ambiente
HML PRD DEV
▾
ℹ️
Esta fase configura o filtro de monitoramento para acompanhar a execução em tempo real. Execute antes de qualquer disparo.
1
Acesse o Workload Monitor e selecione o tipo de servidor: Homologação, Produção ou Desenvolvimento.
2
Clique com o botão direito na listagem e selecione Subscribe with filter...
3
No campo Filter Name, informe o nome da Application.
4
Selecione o campo Application with names e informe: * Nome da RIT * (com asteriscos nas bordas).
5
Acione o botão Save and Subscribe para confirmar.

HML PRD DEV
▾
⚠️
Atenção: Sempre marque a opção Submit application on hold antes de acionar o trigger. Isso evita execução imediata e permite revisar os parâmetros.
1
Acesse a aba Define e selecione o tipo de servidor correspondente.
2
Clique com o botão direito e selecione Download.
3
Em Search string, informe o nome da Application e clique em Ok.
4
Em Event Name, clique com o botão direito sobre a application e selecione Trigger...
5
📋 Confira a descrição da solicitação do ISTM para verificar o que deve ser realizado no Workload.
6
✅ Marque obrigatoriamente a opção: Submit application on hold.
7
Em Select Jobs, selecione os jobs conforme o Checklist da RIT.
8
Preencha os parâmetros conforme a tabela abaixo e acione Ok.
📋 Parâmetros de Trigger
Parâm 1 Número da RIT no ISTM (ex: RIT-00123)
Parâm 2 Servidor indicado no checklist — apenas o prefixo antes do primeiro ponto
Parâm 3 Usuário indicado no checklist
Parâm 4 E-mail do solicitante registrado no ISTM
FASE 03 🚀 Monitor — Configuração e Release
HML PRD DEV
▾
🔶
A requisição aparecerá em LARANJA (Hold). Isso é esperado. Configure todos os jobs antes de fazer o Release.
1
Volte ao Monitor e localize a requisição em laranja (Hold).
2
Para cada Job, clique com o botão direito e selecione Reset Definitions.
3
Informe o Script/Command Name descrito no job do checklist.
4
Informe os Arguments to pass conforme o checklist.
Atenção: sempre separe os argumentos com um espaço entre cada um.
5
✅ Confira se todos os jobs foram preenchidos com Reset Definitions antes de prosseguir.
6
📋 Confira a descrição da solicitação do ISTM para verificar o que deve ser realizado no Workload.
7
Clique com o botão direito na execução e selecione Release Application.
8
Informe o número da RIT no campo de comentário e clique em Ok.

Referência Técnica
Glossário de Termos
A Application ▾
Conjunto de jobs agrupados no Workload que representam um processo de negócio completo. Cada application possui um nome único e é disparada via trigger no módulo Define.
A Arguments to pass ▾
Campo do Reset Definitions onde são informados os argumentos que serão passados ao script durante a execução. Cada argumento deve ser separado por um espaço. Definidos no checklist da RIT.
H Hold (Em espera) ▾
Status em que a application fica após ser disparada com a opção Submit application on hold marcada. A requisição aparece em laranja no Monitor e aguarda configuração e liberação manual antes de executar.
I ISTM ▾
Sistema de gerenciamento de chamados utilizado pela Dataprev para registrar, acompanhar e fechar requisições de TI. É onde as RITs são abertas pelos solicitantes e onde o operador registra a conclusão da execução.
R Release Application ▾
Ação executada via botão direito no Monitor após configurar todos os jobs em Hold. Libera a application para execução efetiva no servidor. É necessário informar o número da RIT no campo de comentário ao confirmar.
R Reset Definitions ▾
Ação aplicada a cada job individualmente no Monitor (Job Avulso) para sobrescrever o script e os argumentos padrão com os valores definidos no checklist da RIT. Deve ser feita em todos os jobs antes do Release.
R Retrieve Spool File ▾
Log de execução completo de um job, acessível via botão direito no Monitor. Deve ser anexado ao comentário do ISTM nos casos de erro, pois contém informações detalhadas sobre a falha para análise técnica.
R RIT ▾
Requisição de TI aberta no ISTM pelo solicitante. Contém as instruções, parâmetros e checklist necessários para a execução do job. O número da RIT é utilizado como referência em toda a operação.
S Script / Command Name ▾
Nome do arquivo de script ou comando que será executado pelo job. Informado no campo de mesmo nome durante o Reset Definitions, conforme descrito no checklist da RIT.
S Subscribe with filter ▾
Função do Monitor que cria um filtro de visualização para acompanhar execuções específicas. Permite monitorar apenas as applications relacionadas à RIT em andamento, sem poluição visual de outros processos.
T Trigger ▾
Ação de disparo de uma application no módulo Define do Workload. Ao acionar o trigger, o operador define quais jobs serão executados e informa os parâmetros necessários para o processamento.
W Workload ▾
Sistema de agendamento e execução de jobs utilizado pela Dataprev. Possui dois módulos principais: Monitor (acompanhamento de execuções em tempo real) e Define (configuração e disparo de applications).

Dúvidas Frequentes
FAQ
❓ Esqueci de marcar "Submit application on hold". O que fazer? Geral ▾
Se a application já foi disparada sem o Hold, ela pode ter iniciado a execução imediatamente. Verifique o status no Monitor. Caso ainda não tenha executado, tente cancelar. Se já estiver rodando, acione o líder ou gerente da equipe para orientação antes de qualquer ação.
❓ A requisição não aparece em laranja no Monitor após o trigger. O que pode ser? Geral ▾
Verifique se o filtro Subscribe with filter está configurado corretamente com o nome da application e a RIT. Confirme também se o servidor selecionado no Monitor é o mesmo utilizado no Define. Se o problema persistir, consulte o líder.
❓ No Job Avulso, preciso fazer Reset Definitions em todos os jobs mesmo que o script seja igual? Avulso ▾
Sim. O Reset Definitions deve ser aplicado em todos os jobs listados no checklist, independentemente de os valores serem iguais. Isso garante que os parâmetros corretos da RIT estejam ativos para esta execução específica.
❓ No Job Eventual, os parâmetros estão como N/A na requisição. Deixo em branco ou coloco "N/A"? Eventual ▾
Deixe o campo em branco. Quando a requisição informa N/A ou não informa nada, o campo de parâmetro deve ficar vazio — não escreva "N/A" no campo, pois isso pode ser interpretado como valor pelo sistema.
❓ Qual a diferença entre Job Avulso e Job Eventual? Geral ▾
O Job Avulso exige Reset Definitions em cada job com script e argumentos fixos do checklist, e os parâmetros do trigger seguem um padrão definido (RIT, servidor, usuário, e-mail). O Job Eventual é mais flexível: os parâmetros vêm da requisição do ISTM (podendo ficar em branco), não exige Reset Definitions e permite visualizar o histórico de execuções via Last # of generations.
❓ O job finalizou com erro. O que incluir no comentário do ISTM? Geral ▾
Use o modelo "Com Erro" disponível na aba ISTM de cada seção. É obrigatório incluir dois itens: (1) os Details do job, obtidos via botão direito → Details no Monitor; e (2) o Retrieve Spool File, que é o log completo da execução. Ambos são necessários para análise da equipe responsável.
❓ Posso fechar o chamado no ISTM antes do job terminar de executar? Geral ▾
Sim, mas use o modelo correto. Se o job ainda está rodando, use o modelo "Em Execução" com o código Processo em execução. Quando finalizar, monitore o resultado no Workload e, se necessário, atualize o chamado com o resultado final.
❓ Como saber qual servidor (HML, PRD ou DEV) usar? Geral ▾
O servidor correto é indicado no checklist da RIT no ISTM. No Job Avulso, o Parâmetro 2 corresponde ao servidor — use apenas o prefixo antes do primeiro ponto do nome. Em caso de dúvida, consulte o líder ou gerente antes de disparar.
